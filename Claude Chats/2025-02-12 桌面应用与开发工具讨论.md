## 桌面应用框架对比

| 框架 | 语言 | 打包体积 | 内存占用 | 特点 |
|------|------|----------|----------|------|
| **Tauri** | Rust + Web前端 | ~2-10 MB | 低 | 最轻量，用系统WebView |
| **Electron** | JS/TS | ~150+ MB | 高 | 生态最成熟，捆绑Chromium |
| **Wails** | Go + Web前端 | ~10-15 MB | 低 | Go版Tauri，用系统WebView |
| **Neutralinojs** | JS/TS | ~2-5 MB | 很低 | 极简，功能有限 |
| **Flutter Desktop** | Dart | ~20-30 MB | 中 | 自绘引擎，跨平台一致 |

## Python GUI 框架对比

| 框架 | 类型 | 轻量程度 | 学习曲线 | 特点 |
|------|------|----------|----------|------|
| **Tkinter** | 原生控件 | 极轻（内置） | 低 | Python 自带，无需安装 |
| **PyQt / PySide** | 原生控件 | 重（~100MB） | 中高 | 功能最强大，Qt 引擎 |
| **Dear PyGui** | GPU渲染 | 轻 | 低 | GPU加速，适合数据可视化 |
| **Flet** | Web渲染 | 中 | 低 | 基于 Flutter，Material Design |
| **CustomTkinter** | 原生控件 | 轻 | 低 | 现代化的 Tkinter |

Python GUI 打包后体积偏大（30-150 MB），因为需要捆绑 Python 解释器 + 依赖库。

## Flutter Desktop

- 跨平台一致性最好，同一套代码跑 Windows/macOS/Linux/iOS/Android/Web
- 自绘引擎（Skia/Impeller），60fps 流畅
- 缺点：原生感缺失、系统集成弱、打包 ~20-30MB、需要学 Dart
- 核心价值是"一套代码全平台"，如果只做桌面端不是最佳选择

## Tauri 为什么这么轻量

核心原因：**不捆绑浏览器引擎**，使用系统自带的 WebView：
- macOS → WKWebView（Safari 内核）
- Windows → WebView2（Edge 内核）
- Linux → WebKitGTK

后端用 Rust 编译成原生机器码，没有运行时开销。

大部分场景不需要写 Rust，Tauri 提供了 JS API 覆盖常见需求（文件操作、对话框、HTTP 请求、通知、剪贴板等）。

## Bun vs Node.js（CLI 工具场景）

| 指标 | Bun | Node.js |
|------|-----|---------|
| 启动速度 | ~5ms | ~30-50ms |
| 打包成单文件 | 内置 `bun build --compile` | 需要 pkg/nexe/sea |
| TypeScript | 原生支持，直接跑 | 需要 tsx/ts-node |
| SQLite | 内置 `bun:sqlite` | 需要装 better-sqlite3 |
| Shell 命令 | 内置 `Bun.$` | 需要 execa/shelljs |

Bun compile 产物 ~50MB（包含 JavaScriptCore 引擎），如果在意体积，CLI 工具最佳选择是 Go（5-10MB）或 Rust（2-5MB）。

本地装了 Bun 的话，直接 `bun run cli.ts` 就行，不用打包。

### Bun 独有内置 API

- `bun:sqlite` — 内置 SQLite
- `bun:test` — 内置测试框架
- `bun:ffi` — 调用 C/Rust 动态库
- `Bun.serve()` — HTTP 服务器
- `Bun.file()` / `Bun.write()` — 文件操作
- `Bun.password` — 密码哈希
- `Bun.$` — Shell 命令

## SQLite

世界上部署量最大的数据库，每部手机、每个浏览器都在用。零配置、零依赖、单文件。适合本地应用数据存储，不适合多服务器并发写入场景。

## Commander（Node.js CLI 框架）

适合给开发者用的 CLI 工具（通过 `npm install -g` 分发），但用户需要安装 Node.js。要给非开发者独立分发，Go/Rust/Bun compile 更合适。
